# Architecture Decision Anti-Patterns

> ADR/RFC 作成の落とし穴、意思決定バイアス、ドキュメント品質の罠、レビュープロセスの失敗

## 1. アーキテクチャ意思決定 7 大アンチパターン

| # | アンチパターン | 問題 | 兆候 | 対策 |
|---|-------------|------|------|------|
| **AD-01** | **Fairy Tale（おとぎ話）** | メリットだけ列挙しデメリットを隠す | 「〇〇は素晴らしい」的な記述、トレードオフ欄が空 | Pros/Cons を同等の詳細度で記述、全主張にエビデンス |
| **AD-02** | **Dummy Alternative（当て馬）** | 本命を選ばせるために非現実的な代替案を並べる | 「自前DB構築 vs PostgreSQL」のような不公平な比較 | 各代替案が実際に実現可能かを検証、独立レビュー |
| **AD-03** | **Mega-ADR（巨大ADR）** | 設計ドキュメント全体を1つのADRに詰め込む | ADR が10ページ超、複数の決定が混在 | 1 ADR = 1 決定、詳細設計は別ドキュメントに分離 |
| **AD-04** | **Sprint（拙速決定）** | 短期的効果のみ考慮し中長期の影響を無視 | 「今動けばいい」「後で直す」が頻出 | 中長期の影響を明示的に記述、6ヶ月後レビュー設定 |
| **AD-05** | **Analysis Paralysis（分析麻痺）** | 完璧な情報を求めて決定を先送り | 2-3スプリント以上決定が停滞、会議が繰り返される | 決定期限を設定、不確実性レベルを明示、段階的決定 |
| **AD-06** | **Cargo Cult（カーゴカルト）** | FAANG等の事例をコンテキスト無視でコピー | 「Googleがやっている」が根拠、自社の規模・制約を無視 | 自社の QAR（品質属性要求）に基づく判断、規模の適合性検証 |
| **AD-07** | **Ghost Decision（幽霊決定）** | 重要な決定が口頭やメールに埋もれ記録されない | 「なぜこうなった？」に誰も答えられない | ADR ログ必須化、決定の DoD（Definition of Done）適用 |

## 2. ADR ドキュメント品質の落とし穴

```
ADR 記述時の具体的な罠:

  Sales Pitch（営業トーク）:
    ❌ 「この技術は革命的で圧倒的に優れている」
    ✅ 「レイテンシ要件 <100ms を満たし、運用コストは月額 $X」
    → 形容詞を排除し、定量的・検証可能な記述に

  Free Lunch Coupon（無料ランチ幻想）:
    ❌ マイクロサービス化のメリットのみ記載
    ✅ デプロイ独立性の獲得 vs 分散トランザクション複雑性の増加
    → 隠れたコスト・長期的な影響を明示的に記述

  Tunnel Vision（トンネル視野）:
    ❌ 開発チームの利便性のみで技術選定
    ✅ 開発・運用・セキュリティ・ユーザー影響を網羅
    → ステークホルダー全体の関心事をチェックリスト化

  Maze（迷路）:
    ❌ ADR のトピックと内容が乖離、脱線が多い
    ✅ 1 ADR = 1 決定、脱線は付録または別 ADR に
    → レビュー時にスコープ逸脱を指摘

  Blueprint in Disguise（偽装設計書）:
    ❌ ADR が権威的な「こうすべき」調の設計書になる
    ✅ 「なぜこの決定に至ったか」のジャーナル形式
    → 決定の経緯と根拠を記録する文書であることを徹底
```

## 3. 意思決定の DoD（Definition of Done）

```
アーキテクチャ意思決定の完了条件（5 要素）:

  1. Evidence（証拠）:
     → 決定を裏付ける技術的根拠・ベンチマーク・PoC結果
     → 「思う」「感じる」ではなく検証データに基づく

  2. Criteria & Alternatives（基準と代替案）:
     → 評価基準を事前に明示（性能/コスト/学習コスト/運用性）
     → 最低2つの実現可能な代替案を評価

  3. Agreement（合意）:
     → 影響を受けるステークホルダーの合意を記録
     → 反対意見があればその理由も記録

  4. Documentation（文書化）:
     → ADR として正式に記録、バージョン管理に含める
     → コンテキスト・決定・結果を明確に分離

  5. Realization & Review Plan（実現・レビュー計画）:
     → 実装方法と検証方法を明示
     → 次回レビュー時期を設定（例: 6ヶ月後）
```

## 4. ADR を書くべきタイミング

```
ADR 作成が必要な場面:

  必須:
    □ フレームワーク・言語・主要ライブラリの選定
    □ アーキテクチャパターンの選択（モノリス/マイクロサービス等）
    □ 重大なトレードオフが存在する決定
    □ 横断的関心事（認証/ログ/エラーハンドリング方針）
    □ 元に戻すコストが高い決定

  推奨:
    □ チーム内で議論が分かれた選択
    □ 将来のステークホルダーが「なぜ？」と問うであろう決定
    □ 過去の決定を覆す・変更する場合

  不要:
    ✗ 実装詳細レベルの判断（変数名、関数構造）
    ✗ 一時的なワークアラウンド（別途チケットで管理）
    ✗ チーム全体に影響しない個人的な技術選択
```

## 5. Atlas との連携

```
Atlas での活用:
  1. PLAN フェーズで AD-01〜07 のスクリーニング
  2. adr-rfc-templates.md と連携した ADR 品質管理
  3. VERIFY フェーズで DoD 5 要素の充足確認
  4. PRESENT フェーズで意思決定の透明性検証

品質ゲート:
  - メリットのみ記載 → Pros/Cons 均等記述必須（AD-01 防止）
  - 非現実的代替案 → 実現可能性検証必須（AD-02 防止）
  - 巨大 ADR → 1 ADR = 1 決定に分割（AD-03 防止）
  - 短期視点のみ → 中長期影響の明示必須（AD-04 防止）
  - 決定停滞 → 期限設定 + 段階的決定（AD-05 防止）
  - コンテキスト無視コピー → 自社 QAR 検証必須（AD-06 防止）
  - 口頭決定 → ADR ログ必須化（AD-07 防止）
```
