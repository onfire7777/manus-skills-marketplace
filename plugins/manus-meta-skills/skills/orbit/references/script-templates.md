# Orbit Script Templates

Purpose: load this first when Orbit must generate or patch loop scripts. It tells you which template file to open next without forcing all executable bodies into context at once.

## Contents

- [Generation decision table](#generation-decision-table)
- [Idempotency notes](#idempotency-notes)

## Generation Decision Table

| Scenario | Generate | Do not generate | Read next |
|----------|----------|-----------------|-----------|
| New loop startup | `bootstrap.sh` | `run-loop.sh` and `notify.sh` separately, because `bootstrap.sh` embeds them | `references/script-template-support.md` |
| Main runner patch | `run-loop.sh` | other scripts unless the request names them | `references/script-template-runner.md` |
| Add verify command | `verify.sh` only | unrelated scripts | `references/script-template-support.md` |
| State recovery | `recover.sh` only | `run-loop.sh` unless recovery changes runner logic | `references/script-template-support.md` |
| Add notification | `notify.sh` only | unrelated scripts | `references/script-template-support.md` |
| Mixed script patch | named targets only | untouched scripts | open only the needed template files |

## Idempotency Notes

- `bootstrap.sh` does not overwrite existing `goal.md` or `progress.md`; it only creates missing artifacts and always refreshes `state.env`.
- `run-loop.sh` is normally generated by `bootstrap.sh`; direct generation is exceptional.
- `recover.sh` is designed to be idempotent because it rebuilds `state.env` from evidence.
- `notify.sh` is always generated, but it is active only when `NOTIFY_ENABLED=true`.
